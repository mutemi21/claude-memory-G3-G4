# STM32 Firmware G4 - Codebase Summary

**MCU:** STM32G071CB (Cortex-M0+)
**Build:** CMake-based, STM32CubeMX-generated project
**Product:** Burn Manufacturing G4 Induction Cooker (Paygo-enabled)

---

## Architecture Overview

The firmware follows an **MVP (Model-View-Presenter)** pattern for the UI layer and a **driver abstraction** pattern for power board communication. The main loop is tick-driven (1ms systick) with a cooperative processing model (`_Tick()` for time-critical work, `_Process()` for main-loop work).

### Data Flow
```
Keypad -> KeypadEvents -> UIView -> UIPresenter (state machine) -> UIModel -> IndCooker -> PowerBoard -> [Highway/CHK/Aifer]Driver
                                                                                              |
                                                                      CookerState -> CookingSession -> SessionAggr -> Flash
```

### `ExternalPeripherals/` - Hardware Abstraction Layer (Custom)

#### `PowerBoard/` - Induction Power Board Communication
The abstraction layer for communicating with the power board over UART.

| File | Purpose |
|------|---------|
| [PowerBoard.h](ExternalPeripherals/PowerBoard/Inc/PowerBoard.h) | **Central interface.** Defines `CookerPower_e` (power levels 0-2000W), `CookerError_e` (all error codes), `CookerPotPresence_e`, `CookerPowerBoard_e` (AIFER/HIGHWAY/CHK), and callback types (`StatusCallback`, `ErrorCallback`, `TemperatureCallback`, `ActualPowerCallback`). |
| [PowerBoard.c](ExternalPeripherals/PowerBoard/Src/PowerBoard.c) | **Dispatcher.** Routes `PowerBoard_*()` calls to the correct driver based on `PowerBoardType` (currently hardcoded to HIGHWAY). |
| [HighwayDriver.h](ExternalPeripherals/PowerBoard/Inc/HighwayDriver.h) / [.c](ExternalPeripherals/PowerBoard/Src/HighwayDriver.c) | **Highway power board driver.** UART protocol: sends control bytes for power levels, parses response frames for status/error/temperature. `SetLevel()` triggers `StatusCallback`. `PowerOff()` does NOT trigger callback. |
| [CHKDriver.h](ExternalPeripherals/PowerBoard/Inc/CHKDriver.h) / [.c](ExternalPeripherals/PowerBoard/Src/CHKDriver.c) | **CHK power board driver.** Different UART protocol. Note: `PowerOff()` DOES trigger `StatusCallback` (unlike Highway). |
| [AiferDriver.h](ExternalPeripherals/PowerBoard/Inc/AiferDriver.h) / [.c](ExternalPeripherals/PowerBoard/Src/AiferDriver.c) | **Aifer power board driver.** Third power board variant. |

## Key Callback Chain (Power On Example)

1. User presses ON button -> `KeypadEvents` fires `UIView_UpdateWithKeyPadEvent()`
2. `UIView` stores event -> `UIPresenter_Process()` reads it
3. `UIPresenter` runs action function -> calls `UIModel_PowerOnPowerBoard()`
4. `UIModel` calls `IndCooker_PowerOn()` -> `IndCooker` calls `PowerBoard_SetLevel()`
5. `PowerBoard` dispatches to `HighwayDriver_SetLevel()` -> sends UART command
6. Highway board responds -> `HighwayDriver_Process()` parses response
7. `StatusCallback` fires -> `IndCooker`'s `PowerBoardStatusCb()` -> `CookerState_On()`
8. `CookerState_On()` starts `CookingSession_Start()` -> begins power accumulation

## Key Callback Chain (Error Example)

1. Power board reports error -> driver parses error from UART response
2. `ErrorCallback` fires -> `IndCooker`'s `PowerBoardErrorCb()`
3. `IndCooker` calls `UIPresenter_HandleError()` -> UI transitions to error display state
